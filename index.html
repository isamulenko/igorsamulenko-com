Data Map Search 2.0

Config-driven faceted search for an enterprise data discovery platform serving 5,000+ data workers.

12xFaster queries
10%Precision gain
5K+Daily users
ReactTypeScriptREST APICI/CD

The problem: Data workers across the enterprise were abandoning search queries and asking colleagues on Slack instead. Queries took 3-5 seconds. Precision was poor.

What I built: I led a 6-person team to redesign the search from scratch. The core architectural decision: make the entire faceted search UI config-driven. Each facet type is defined in JSON configuration. Adding a new filter means adding a config entry, not writing new code. This cost two extra sprints upfront but paid for itself within the first quarter as 10+ new filter types were added through config alone.

The result: Queries dropped from 3-5 seconds to 200-400ms. Two of the three engineers I mentored were promoted within a year.

Part of Intuit's Data Mesh initiative, featured at AWS re:Invent and in Intuit's engineering blog.

Lineage UI Performance

Took a data lineage visualization from completely unusable to instant response.

5-6xFaster loads
43%Less memory
0Freeze time
ReactReduxVirtual ScrollingProfiling

The problem: When users opened large data lineage graphs (2,500+ nodes), the app froze for up to 3 minutes. Memory spiked to 700MB. People were force-quitting browsers. 500+ users were affected daily.

The wrong assumption: I initially assumed the backend API was the bottleneck. Spent two days profiling the wrong layer. The backend was fine.

The real fix: Chrome DevTools revealed the frontend was making 3 API calls when only 2 were needed, rendering all 2,500+ rows without virtualization, and duplicating state across components. I eliminated the redundant call, parallelized the remaining two, migrated to virtual scrolling, introduced a shared Redux store, and wrote 24 regression tests.

The lesson: Don't let intuition skip diagnostics. Profile end-to-end first. This became our team's documented best practice for performance debugging.

Frontend Health Scorecard

Built an observability framework measuring frontend maturity across 13 plugins spanning multiple teams.

80%Coverage
13Plugins assessed
$10K+OpEx savings found
RUM/FCICode CoverageStorybookCI/CD

The problem: Frontend technical debt was invisible. No consistent way to measure RUM adoption, code coverage, test infrastructure, or documentation across teams. Teams couldn't advocate for remediation time because they couldn't quantify the problem.

What I built: A phased scorecard framework. Phase 1 established baselines without requiring immediate fixes, reducing resistance. Phase 2 introduced prioritized targets. I framed it as a tool that helps teams make the case for tech debt time, not a stick to beat them with.

The result: 80% coverage across all frontend plugins. $10K+ in OpEx savings identified. Teams that were initially resistant adopted the framework.

How I work

I get things moving. When requirements are vague, I don't wait for clarity to arrive — I dig in, talk to the right people, prototype something tangible, and let that drive the conversation forward. Ambiguity is just a problem to solve, not a reason to stall.

I write the spec, build the first component, and hand the team a clear path. I like giving people direction through working code and well-structured technical documents — not abstract plans. When someone picks up my work, the patterns should make the next decision obvious.

Frontend is where I come alive. I care deeply about component architecture, performance, and that moment when a UI just clicks — when it's fast, polished, and people notice. I've spent 15 years chasing that feeling across React, TypeScript, and large-scale design systems.

I pick up what I need to. I started my career writing Java backends, then moved through ActionScript, AngularJS, and landed on React and TypeScript — each shift because that's where the most interesting problems were. New framework, unfamiliar domain, a tool that didn't exist last month — I learn fast and start contributing before the onboarding doc says I should.

I ship. Scope gets trimmed, edge cases get filed for later, but the thing goes out the door and it works. Then we iterate.
